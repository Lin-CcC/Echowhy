# Echowhy V1.5 插入式卡片统一规范（整合定稿版）

## 版本

`v1.0`

## 文档目标

本规范用于统一 `My Question`、`Feedback`、`Source` 三种插入式卡片的：

- 视觉层级
- 双主题样式
- 拖拽与插入行为
- 折叠与阅读体验
- 后续实现边界

这份文档基于两份现有 PRD 的共同有效部分整合而成，并补齐了当前产品实现中真正需要落地的交互细节，目标不是“设计概念好看”，而是“前端实现稳定、阅读感受清晰、扩展时不打架”。

---

## 一、整合后的核心判断

### 1. 两份 PRD 的共识

两份 PRD 在以下判断上是一致的，这些内容直接确认为最终方向：

- 三种卡片必须被用户一眼区分
- 区分方式不能依赖花哨装饰，而应依赖少量稳定的视觉信号
- 差异化的核心手段是：
  - 左侧状态色条
  - 背景透明度层级
  - 内部排版权重差异
- 不应重做整张卡片结构
- 不应引入额外杂色、额外组件或卡通化图标
- `My Question` 是最重要的前景层
- `Feedback` 是指导层
- `Source` 是参考层

### 2. 两份 PRD 的问题

这两份文档里有一些地方需要纠偏，不能直接照抄：

- 豆包版一边说“禁止新增图标”，一边又在折叠态示例里使用了 emoji，这在规范上自相矛盾
- Gemini 版把折叠态写成 `[ { } ]` / `[ ⚡ ]` / `[ ? ]`，方向比 emoji 好，但仍然偏“标签化”，不够贴近现在产品已经形成的极简风格
- 两份文档都主要讨论“卡片长什么样”，但对“卡片插入到哪里”“卡片之间怎么继续插”“拖拽命中边界怎么控制”描述不够
- 两份文档都没有把“正文可插、卡片本体禁插、卡片上下可插”这种真实实现约束写清楚

### 3. 本次定稿的补强原则

因此，这份统一规范在保留两边精华的同时，新增三个必须落地的维度：

- 插入位规范
- 层级透明度规范
- 折叠态和轻量控制规范

---

## 二、三种卡片的语义定位

| 卡片类型 | 语义角色 | 用户心智 | 视觉层级 |
| :-- | :-- | :-- | :-- |
| `My Question` | 用户主动提出的问题 | “这是我正在思考的东西” | 最高 |
| `Feedback` | 系统对回答的评估与引导 | “这是系统给我的纠偏” | 中间 |
| `Source` | 参考资料或证据片段 | “这是可供对照的材料” | 最低 |

### 最终原则

- `My Question` 负责承接用户意图，必须最靠前
- `Feedback` 负责帮助用户修正表达，必须清楚但不能喧宾夺主
- `Source` 负责作为证据存在，必须稳定、克制、低刺激

---

## 三、共享骨架规范

三种卡片必须共享同一套基础骨架，避免页面碎片化。

### 1. 结构骨架

所有插入卡片默认共享：

- 相同的外边距节奏
- 相同的左侧纵向结构
- 相同的基础圆角级别
- 相同的边框精度
- 相同的关闭按钮位置
- 相同的内部标题区与内容区分层

### 2. 禁止差异化的部分

以下部分三种卡片不能各搞一套：

- 圆角大小
- 阴影强度
- 外轮廓边框粗细
- 关闭按钮形态
- 内容区左右内边距
- 卡片与正文之间的默认垂直间距

### 3. 允许差异化的部分

只允许通过以下三个维度做差异化：

- 左侧色条颜色
- 背景材质与透明度
- 内部排版强弱

---

## 四、双主题视觉规范

## 4.1 暗色主题

### My Question

- 左侧色条：`cyan-blue` 倾向的高亮蓝
- 背景材质：最实的深色毛玻璃
- 建议表现：
  - `border-l-2`
  - `bg-black/45` 到 `bg-slate-950/45`
  - `backdrop-blur-sm`

### Feedback

- 左侧色条：低饱和琥珀色
- 背景材质：比 My Question 更透
- 建议表现：
  - `bg-amber-950/14` 到 `bg-black/28`
  - 不允许高饱和橙块

### Source

- 左侧色条：青色
- 背景材质：三者中最透
- 建议表现：
  - `bg-cyan-950/10` 到 `bg-black/22`
  - 代码块内层可再加极轻的冷色底

### 暗色主题层级关系

- `My Question` 最实
- `Feedback` 次之
- `Source` 最轻

这不是为了“看出三张卡”，而是为了让用户自然把注意力先放在自己的问题上。

## 4.2 亮色主题

### My Question

- 左侧色条：蓝色
- 背景材质：最接近纯白，但不要完全死白
- 建议表现：
  - `bg-white/82`
  - 或非常轻的冷白玻璃

### Feedback

- 左侧色条：琥珀色
- 背景材质：非常淡的暖色雾面
- 建议表现：
  - `bg-amber-50/34`

### Source

- 左侧色条：青色
- 背景材质：非常淡的灰蓝或冷白
- 建议表现：
  - `bg-slate-50/58`
  - 不要把源码卡片做成比正文还亮

### 亮色主题层级关系

- 亮色主题同样保持：
  - `My Question` 最清晰
  - `Feedback` 中间
  - `Source` 最轻

## 五、内部排版差异化规范

## 5.1 My Question

### 标签

- 左上角标签使用：
  - `MY QUESTION`
- 小号、大写、字距拉开
- 颜色跟左侧色条同系

### 主问题

- 问题文字是卡片主体
- 需要是三类卡片中最容易被看到的正文
- 不需要额外大号，只需要比另外两种更清楚

### 输入区

- 输入文本与下方横线的垂直间距必须收紧
- 文字不能“飘在横线上方”
- 输入区的可编辑感来自排版和细线，不靠色块

### 操作按钮

- `[ CHECK ]` 仍然是主操作
- `cancel / close / delete` 等次级操作不应比 `[ CHECK ]` 更抢眼

## 5.2 Feedback

### 标签

- 左上角标签使用：
  - `FEEDBACK`
  - 或 `FEEDBACK INSERT`
- 使用琥珀色小号大写字

### 分数信息

- 分数可以强化，但不要变成徽章系统
- 推荐：
  - 数字加粗
  - 评级名和分值形成一组
- 不把分数抛到右上角单独做成药丸主视觉

### 内容组织

- `What landed well`
- `What feels unclear`
- `What’s still missing`
- `A good next step`

这些结构应该保持清晰，但不应新增额外装饰图标来撑层次。

## 5.3 Source

### 标签

- 左上角标签使用：
  - `SOURCE`
  - 或 `SOURCE INSERT`

### 文件信息

- 文件名 / 路径 / 行号必须比正文更冷静
- 文件路径字号比普通正文小一档
- 行号和路径不能太亮，避免比代码内容更显眼

### 代码区

- 必须使用等宽字体
- 字号比正文小 1px 是合理的
- 代码容器内部允许极淡底色
- 代码高亮保持现有高亮逻辑，不另造一套视觉语言

## 六、拖拽与插入规范

这是两份 PRD 都没写透，但实际最重要的部分。

## 6.1 拖拽中的视觉反馈

拖拽卡片时：

- 卡片整体可以轻微缩小
- 透明度略降
- 但左侧色条必须保留

### 插入提示线颜色

插入提示线必须跟拖拽对象绑定：

- 拖 `My Question`：蓝色 / 青蓝
- 拖 `Feedback`：琥珀色
- 拖 `Source`：青色

### 提示线状态

- 普通命中：细线
- 即将落位：略变亮、略变粗
- 不要加粗暴发光

## 6.2 插入命中规则

### 正确规则

- 正文段落可以插入
- 卡片本体不能插入
- 卡片上方可以插入
- 卡片下方可以插入

### 错误规则

以下都是需要禁止的：

- 把问题直接插进卡片内部
- 把提示线显示在卡片正文中间
- 让整块卡片容器都成为可插入热区

### 最终命中模型

建议把插入区分为三种：

- 段落插槽
- 卡片前插槽
- 卡片后插槽

这样结构清晰，也最不容易反复出 bug。

## 6.3 插入后的行为

插入之后：

- 卡片自动进入内容流
- 宽度响应式适配正文栏
- 与上下正文保持统一节奏
- 不破坏原有阅读宽度

## 七、折叠态规范

两份 PRD 在这里都有好点子，但我建议收成更克制的版本。

### 折叠态原则

- 仍然要一眼看出卡片类型
- 但不要依赖 emoji
- 也不要依赖过度装饰的专属图标

### 最终建议

#### My Question 折叠态

- 左侧保留蓝色色条
- 主内容：
  - 问题标题
  - 可附带极轻的状态词，如 `draft / checked`

#### Feedback 折叠态

- 左侧保留琥珀色条
- 主内容：
  - 评级 + 分数
  - 如：`WEAK / 18`

#### Source 折叠态

- 左侧保留青色条
- 主内容：
  - 文件名 + 行号
  - 如：`AuthService.java : 18-22`

### 为什么不用 emoji

- 和产品现有极简深空风格不兼容
- 会破坏通透感
- 一旦进入暗色模式，会显得像 UI 外来物

---

## 八、层级与沉浸感规则

### 1. 卡片不能打破星空主题

三类卡片都应当像“漂浮在阅读流中的薄层”，而不是厚重的补丁。

### 2. 背景处理原则

- 不使用不透明大色块
- 不用重阴影压死背景
- 透明度只能帮助分层，不能破坏星空通透感

### 3. Source 卡最容易做丑

所以特别规定：

- Source 卡的存在感一定要比 My Question 低
- 但不能低到不可读
- 它是证据，不是主角

---

## 九、实现建议

结合当前代码结构，建议实现时遵守以下方式：

### 1. 基础骨架 class 抽离

抽一个卡片基础 class，例如：

- 外容器
- 标题区
- 内容区
- 关闭按钮区

然后三种卡片通过 modifier class 注入：

- `question`
- `feedback`
- `source`

### 2. 差异化只通过 modifier 做

不要为三种卡片分别重新造三套 DOM。

### 3. 插槽和卡片分离

插槽必须是独立结构，不能和卡片本体共用命中区。

### 4. 折叠态与展开态共用主题信号

即：

- 左侧色条颜色不变
- 字体层级不变
- 仅内容密度变化

---

## 十、最终定稿结论

### 本次采用的精华

从两份 PRD 中保留的有效部分：

- 三卡分层思路
- 左侧色条作为主识别信号
- 透明度梯度而不是装饰堆砌
- 代码卡使用等宽字体与更轻背景
- 拖拽提示线颜色跟随卡片类型
- 通过排版而非重构 DOM 做差异化

### 本次去掉的糟粕

- emoji 折叠态
- 自相矛盾的“禁止图标但示例全是图标”
- 过于概念化、没有实现价值的装饰描述
- 对插入命中区域没有约束的模糊写法

### 本次新增的必要补充

- 卡片本体禁插、卡片上下可插
- 插槽与卡片结构解耦
- 折叠态改为文字主导而不是 emoji 主导
- Source 卡必须“更轻”，但不能“更弱到看不见”

---

## 十一、验收标准

### 视觉验收

- 用户在 0.5 秒内能区分三类卡片
- `My Question` 永远是最自然的视觉焦点
- `Feedback` 清晰但不喧宾夺主
- `Source` 低调、可读、稳定

### 插入验收

- 正文可插
- 卡片上方可插
- 卡片下方可插
- 卡片本体不可插
- 提示线颜色与拖拽对象一致

### 折叠验收

- 折叠后仍能分辨卡片类型
- 不依赖 emoji
- 不依赖多余图标

### 主题验收

- 暗色主题下三种卡片有清晰层级
- 亮色主题下三种卡片仍然保留差异
- 所有卡片与星空/数学主题背景不打架

---

## 十二、推荐落地优先级

1. 统一三类卡片的基础骨架
2. 落地三类卡片的双主题 modifier
3. 统一拖拽提示线颜色
4. 修正插入命中模型
5. 统一折叠态
6. 最后再做字号、字重、透明度微调

这份规范可以作为后续插入式卡片的唯一参考版本，后续实现以它为准，不再同时参考两份原始 PRD。
